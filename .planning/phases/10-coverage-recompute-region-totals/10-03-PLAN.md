## DECISION-7 RECONCILIATION (explicit — read this)
10-CONTEXT decision 7 says "offline tool = pyosmium/osmium, accept the Python-env dependency." **This plan does NOT use pyosmium.** The 10-RESEARCH deep-dive (HIGH confidence, verified from source) found that `tool/osm_pipeline/` is ALREADY a complete Dart-native OSM PBF pipeline that does every required operation: streaming PBF read, admin multipolygon assembly with L9 already in `kTargetAdminLevels = {2,4,6,8,9,10}`, way→region spatial join at Germany scale (`buildWayAdminJoin`), Kfz filter (`kKfzHighwayTags`, bit-for-bit identical to the runtime `kfzHighwayClasses`), and haversine length (`haversineMeters`). Decision 7's text predates knowing the Dart pipeline's full capability.

**Adopt the Dart-native Stage-H approach (research recommendation (a)).** Rationale:
- Reuses the existing, verified, Germany-scale pipeline — one source of truth, one PBF download.
- The admin bundle and totals table come from the SAME run and SAME osm_ids — this is exactly what invariant 5 requires, and hard to guarantee if polygons came from Overpass/one tool and totals from another.
- No new Python runtime/env dependency on the dev machine; no new spatial libs.
- The Kfz allowlist is already applied at Stage B (`source='kfz'`), so totals/matcher parity is guaranteed by construction.
No blocker was found. Decision 7 is treated as "use an offline PBF tool"; the Dart pipeline is that tool. This reconciliation is surfaced here so the user sees the deviation and its rationale.

Purpose: Correct, instant, offline per-region totals + Ortsteil granularity, atomically consistent with what the pill can name.
