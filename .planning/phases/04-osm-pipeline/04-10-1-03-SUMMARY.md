# Phase 4 Plan 04-10.1 Wave 3: perWay R-Tree Default + CLI Flag

**One-liner:** R-Tree default flipped from perSegment (555 920 rows on Berlin at Wave 1 baseline / 322 184 rows post-Feldweg-drop) to perWay (91 707 rows = one row per Kfz way). New `--rtree-granularity=perSegment|perWay` CLI flag preserves the old behavior as an opt-in fallback. Berlin osm.sqlite shrank 49.06 MiB → 30.77 MiB (−37% monotonic vs Wave 2); the plan's aspirational <25 MB hard gate is NOT met and is documented as a plan-projection error rather than a code defect.

### Hard-gate miss: Berlin osm.sqlite <25 MB not met (30.77 MiB delivered)

Plan spec: "osm.sqlite bytes < **25 MB** on Berlin — this is the plan's hard gate". Delivered: 32 264 192 B (30.77 MiB).

**Root cause: plan-projection error, not code defect.**

The perWay flip achieved its research-§4-projected R-Tree cluster shrinkage almost exactly (71% vs the projected ~65%). The R-Tree cluster shrunk from 25.64 MiB (perSegment / Wave 2) to 7.35 MiB (perWay / Wave 3) — an absolute saving of 18.29 MiB. But the residual Wave-3-invariant bytes are dominated by:

- `ways` inline LineString-WKB geometry: 13.31 MiB
- `way_admin` + `idx_way_admin_region`: 5.77 MiB combined
- `idx_ways_highway` + `idx_ways_source_counting`: 3.40 MiB combined

Those three groups alone sum to **22.48 MiB** — with any non-zero R-Tree cluster + admin_regions + metadata + schema, the DB cannot drop below ~24-25 MB without introducing new levers outside Wave 3's scope (varint coord deltas, dropping way-index tables, admin coarsening). Research §7 anticipates this: line 458 states "osm.sqlite achievable: ~1.4-1.6 GB with just E.1 + E.2 (no format changes)". Extrapolated to Berlin's 91 707 Kfz ways, that projection lands at ~32-37 MiB — matching what we delivered.

**Investigation as required by the prompt ("If Berlin doesn't hit the <25 MB gate: STOP and investigate — projection may be off; don't just note it in SUMMARY and move on"):**

1. Ran a `dbstat` breakdown of the final osm.sqlite (table above).
2. Compared each object's bytes against the perSegment baseline — the R-Tree cluster contribution alone accounts for the entire Wave 3 delta.
3. Verified perSegment produces the exact Wave 2 byte count (51 445 760 B), ruling out any accidental data drift.
4. Verified `segment_idx` sentinel and row-count invariants — Wave 3 semantics are correct.

**Recommendation for Sub-Phase 04-10.1 close-out (Wave 5 or a Wave 6):**

- Update the plan's estimate of Berlin post-Wave-3 size to ~30-33 MiB (matches research §7's per-way byte model) so future waves inherit an accurate baseline.
- The <25 MB target may still be reachable in Wave 4/5 via varint delta encoding of `geometry_wkb` (research §7 projects −50% on the ways table alone → ~7 MiB saving → ~24 MB total). Not in Wave 3 scope.
- Alternative: renegotiate the Berlin per-wave target the same way SC4 was relaxed 200 → 800 MB in 04-06. The Germany projection under E.1+E.2 (1.6 GB per research §7 line 440) is what matters for SC4; Berlin's 30 MB is comfortably competitive.

**No code changes made in response to this miss** — the code delivers exactly the semantics the plan specified, and no in-scope Wave 3 lever could close the ~5.7 MB gap without touching Wave 4/5 territory (geometry encoding, additional index drops).
