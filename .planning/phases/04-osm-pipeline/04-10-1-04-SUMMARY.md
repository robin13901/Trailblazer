# Phase 4 Plan 04-10.1 Wave 4: Stage D Isolate Parallelism + `--workers=N` CLI

**One-liner:** Stage D (way_admin join) now runs across N worker isolates behind a new `--workers=N` CLI flag. Berlin measurement: Stage D wall-clock 36s → 14s at workers=6 (**2.57× speedup** — clears the plan's `T_parallel ≤ T_serial/2` gate). All four osm.sqlite tables (ways / ways_rtree_lookup / way_admin / admin_regions) are per-table `.dump | sha256sum` identical AND COUNT-identical between `--workers=1` and `--workers=6` — content-identical output invariant confirmed. Full sub-package: 239/239 tests green (was 233; +6 new multi-worker correctness tests).

## What shipped

### Code (6 files, 1 new + 5 modified)

**`tool/osm_pipeline/lib/intersect/way_admin_join_isolate.dart` (NEW, ~320 lines)**

- Top-level `Future<void> wayAdminJoinWorkerEntry(WorkerArgs)` — the isolate entry point. Windows requires top-level or static entry points for `Isolate.spawn` (research §5.2 pitfall). Not a class method, not a closure.
- Marshalable payloads (all fields either primitives, `String`, `Uint8List`, `SendPort`, or `List<int>`):
  - `WorkerArgs(workerId, scratchDbPath, wayIds, sendPort, flushEvery)`
  - `WayAdminResult(wayId, regionId, adminLevel, fractionStart, fractionEnd)` — 5-tuple for one way_admin_raw row.
  - `WorkerBatch(workerId, tuples, tickDelta)` — batched flush envelope.
  - `WorkerDone(workerId)` — sentinel posted as the final message before the isolate returns.
- Worker flow: `sqlite3.open(scratchDbPath, mode: OpenMode.readOnly)` inside the isolate → `PRAGMA busy_timeout=30000` → load admin_regions once → prepare `SELECT lat, lng FROM nodes_raw WHERE id=?` + `SELECT node_ids FROM ways_raw WHERE id=?` → iterate the partitioned way ids → clip against every level's admins → append `WayAdminResult` to a local batch → flush envelope every `flushEvery=5000` tuples via `sendPort.send(WorkerBatch(...))` → drain final batch → post `WorkerDone` → return (Dart auto-fires `onExit(null)`).
- `kAdminLevels` + `decodeMultiPolygonWkb` + `decodeNodeIds` shared with the serial path via explicit `show` imports. `_loadAdmins` / `_Bbox` / `_AdminEntry` / `_bboxOfLine` / `_bboxOverlap` inlined per Plan Task 1 §Note (avoid promoting private helpers to a shared surface; simpler ownership).
- **Test-only poison sentinel:** negative `way_id` (never appears in real OSM data — OSM IDs are always positive int64) makes the worker throw `StateError` from its entry point. Exercised by the crash-escalation test in Task 3.

**`tool/osm_pipeline/lib/intersect/way_admin_join.dart`**

- `buildWayAdminJoin` signature: `Future<WayAdminJoinStats> buildWayAdminJoin(ScratchDb, {int workers = 1})`. Dispatch on `workers`: `1` → `_runSerial` (verbatim pre-Wave-4 code except for the new `.workers` stats field), `>1` → `_runParallel`.
- Clamping: values are clamped to `[1, 16]` at entry — `workers=0 → 1`, `workers=32 → 16`. No error thrown; the CLI/orchestrator layers do the "auto-default" pick.
- `_runParallel` flow:
  1. Read all Kfz way ids via `SELECT id FROM ways_raw WHERE source='kfz' ORDER BY id;` — deterministic partition input.
  2. Round-robin partition: worker `i` gets ids where `index % N == i`. Balances load if geometries cluster by contiguous id ranges.
  3. Coordinator `ReceivePort` + `ProgressLogger('Stage D', total: totalKfz, unit: 'ways')` + coordinator-local `Stopwatch swElapsed`.
  4. Prepare **plain** `INSERT INTO way_admin_raw ... VALUES(?,...,?)` — NOT `INSERT OR IGNORE`. Partition-by-way_id guarantees no two workers touch the same way, so any PK collision is a real bug (fail-loud). Serial path keeps `OR IGNORE` for pre-Wave-4 backwards compat.
  5. Spawn each worker via `Isolate.spawn<WorkerArgs>(wayAdminJoinWorkerEntry, args, errorsAreFatal: true, onError: rp.sendPort, onExit: rp.sendPort)`. Retain each `Isolate` handle in a `List<Isolate> spawned`.
  6. Message loop matches Dart's canonical shapes:
     - `WorkerBatch` → for each tuple execute INSERT (any exception → `failFast`), then `progress.absorb(WorkerTick(workerId, tickDelta, swElapsed.elapsedMilliseconds))`.
     - `WorkerDone` → increment `doneCount`. When `doneCount == workers` → `COMMIT`, then `completer.complete()`.
     - `List<Object?>` with `length == 2` → Dart canonical `onError` (`[errorString, stackTraceString]`) → `failFast('Stage D worker crashed: ...', err, st)`.
     - `null` → Dart canonical `onExit`. Count via `exitCount++`. If `exitCount > doneCount` → escalate ("worker exited without WorkerDone signal"). Also drives the drain-until-all-exited invariant.
     - Unknown type → defensive ignore (do not crash the coordinator on stray messages).
  7. `failFast(message, [cause, st])` is **idempotent**: multiple `onError` messages arriving in quick succession collapse to one rollback + throw. Actions: kill all `spawned` isolates via `iso.kill(priority: Isolate.beforeNextEvent)` → `ROLLBACK` (secondary rollback errors swallowed) → `completer.completeError(PipelineRuntimeError(...))`.
  8. Outer try/finally drains onExit signals until `allExited.future` (bounded by 3s timeout) — sqlite3 native FDs held by workers must be released before the coordinator's caller can safely delete the scratch temp dir on Windows.
- `WayAdminJoinStats` gains `.workers: int`. `.candidatePairsProbed` is `-1` in parallel mode (probe counts aren't aggregated back to keep SendPort payload minimal — the field is a perf-tuning hint, not a correctness invariant).

### Tests (test/intersect/way_admin_join_test.dart — 5 existing + 6 new = 11 total, all green)

The 5 pre-Wave-4 tests were converted to `async` and now `await buildWayAdminJoin(...)`. They cover the serial fast-path (default `workers=1`).

New group `'multi-worker correctness'` adds 6 tests:

3. **`workers=32 clamps to 16`** — passing 32 spawns 16 workers even with only 6 ways in the fixture (10 workers get empty partitions; coordinator still waits for their `WorkerDone`).

## Deviations from Plan

### Auto-fixed / minor edits (Rule 1-3)

**1. [Rule 1 — Bug] Coordinator drain wedged on trailing `onExit(null)` after `completer.complete()`**

- **Found during:** Task 3, workers=32 clamp test — added a 3s pause per test with all-workers-exit drain hitting its timeout.
- **Root cause:** My initial coordinator's message-loop early-returned on `if (completer.isCompleted) return;` before the `null` case, so trailing `onExit` signals arriving after the completer resolved (which is normal — `WorkerDone` completes the coordinator; then Dart still fires `onExit` for each isolate a microtask later) were dropped. `exitCount` therefore never reached `workers`, and `allExited.future.timeout(3s)` always exhausted its full window.
- **Fix:** Moved the `msg == null` branch above the early-return guard. Late-arriving `onExit` signals now always count, regardless of coordinator resolution state. Ordering invariant `exitCount > doneCount ⇒ unclean` still holds — WorkerDone is always sent BEFORE the isolate returns, and per-SendPort FIFO guarantees WorkerDone arrives before its owner's null.
- **Files modified:** `lib/intersect/way_admin_join.dart`.
- **Commit:** rolled into `e6ea601`.

**2. [Rule 2 — Missing critical] `Isolate.kill(priority: Isolate.immediate)` leaked sqlite3 FDs on Windows**

- **Found during:** Task 3, poison-way-id crash test — `scratch.close(deleteFile: true)` in tearDown failed with Windows `errno=32` ("used by another process") when peer workers were killed mid-DB-read.
- **Root cause:** `Isolate.immediate` terminates the isolate before its `finally` blocks (which `db.dispose()` the read-only sqlite3 handle) run. The native FD stays open until the OS reaps it seconds later, and Windows refuses to `deleteSync` an in-use file.
- **Fix:** Switched to `Isolate.kill(priority: Isolate.beforeNextEvent)` (the default) — peers get a chance to finish their current statement and run `finally` blocks. Combined with the drain-until-all-exited invariant (§Rule 1 above), the temp-dir cleanup path became reliable enough for the correctness/COUNT-parity tests. The poison-way-id test itself uses a locally-scoped scratch + best-effort tearDown for defensive resilience since 4-way peer kills under a stressed Windows CI runner can occasionally still race.
- **Files modified:** `lib/intersect/way_admin_join.dart`, `test/intersect/way_admin_join_test.dart`.
- **Commit:** rolled into `e6ea601`.

**3. [Rule 2 — Missing critical] Worker read-only `busy_timeout=30000ms` added**

- **Found during:** Task 1 review before Task 2 wiring.
- **Reason:** Coordinator holds a brief EXCLUSIVE lock during each INSERT into `way_admin_raw` (single writer, journal_mode=OFF). Workers reading `nodes_raw` / `ways_raw` / `admin_regions_raw` will race that lock. Without `busy_timeout`, worker SELECTs would immediately fail with SQLITE_BUSY under contention — a fragile setup that would only manifest at Germany scale.
- **Fix:** `db.execute('PRAGMA busy_timeout = 30000;')` immediately after `sqlite3.open(scratchDbPath, mode: OpenMode.readOnly)` in the worker entry.
- **Files modified:** `lib/intersect/way_admin_join_isolate.dart`.
- **Commit:** rolled into `ada12e6`.
