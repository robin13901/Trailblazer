## Deviations from Plan

**None material.** Two minor implementation choices worth noting:

1. **Emit sink injection instead of stderr capture.** The plan said "redirect stderr in the test via `IOOverrides.runZoned` OR wrap Logger in a captured sink — pick whichever your project convention prefers." No prior test in `tool/osm_pipeline/test/` uses either pattern (grep-verified). Chose the sink-injection route: `ProgressLogger(..., emit: sink.add)`. Production defaults to `Logger.info`, so behavior is identical; tests avoid stderr-redirection ceremony. Class is now trivially testable without touching global state.

2. **Total-count SELECT is inline (not pre-computed).** For Stages D + E the plan noted `COUNT(*)` should be cheap. Confirmed on Berlin: no measurable delay. If Germany reveals slowness (unlikely — the `ways_raw` index handles it), we cache the count as a follow-up per the plan's Deviations §. Not needed at Berlin scale.

## Decisions Made (candidates for STATE.md)

- **D-04-10-1-01-A** — ProgressLogger's SendPort ingestion (`absorb(WorkerTick)`) lives in the emitter API from Wave 1. Wave 4 (Stage D isolates) wires workers → coordinator → `absorb(msg)` without refactoring the class. `WorkerTick(workerId, deltaTicks, elapsedMs)` is the marshalable message shape; `deltaTicks` is per-tick delta (not cumulative) so N workers contribute correctly to a single aggregate `_done` counter.
- **D-04-10-1-01-B** — `total=0` is the sentinel for "unknown total up front" — Stage B pass A, Stage C pass A, Stage F.1 water/labels PBF passes all use it. Emitter falls back to rate-only format (`{stage} progress: {done} {unit} — {rate}/s (total unknown)`).
- **D-04-10-1-01-C** — Tippecanoe stdout/stderr lines are pass-through with a `[Stage F.2]` prefix — no attempt to parse tippecanoe's own progress format. Verified end-to-end on Berlin: 60+ `[warn] [Stage F.2] …` lines from tippecanoe's own progress bar landed in the log correctly.
- **D-04-10-1-01-D** — Emit sink is injectable in the `ProgressLogger` constructor (`emit: void Function(String)?`, defaults to `Logger.info`). Enables clean unit tests without stderr redirection; keeps production output routed through the existing `Logger` API surface.
- **D-04-10-1-01-E** — `Logger` static class (23 lines) stays untouched. `ProgressLogger` is a sibling stateful helper in `lib/cli/progress_logger.dart`, not an extension of `logger.dart`. Rationale: many call sites depend on `Logger.info/warn/error`; migrating them all is not this plan's scope.

## Post-close corrective fix (2026-07-07)

Two Wave 1 gaps surfaced during the Germany Wave 5 run and were patched
after this plan closed. Both live inside the 04-10-1-01 scope (progress
logging observability + Wave 1 emitter shape), so they are documented
here rather than as a new SUMMARY.

### Gap 1 — Stage B integrity check had no ProgressLogger tick

**Symptom:** Berlin (91K ways) finished the Stage B post-pass integrity
check in milliseconds; Germany runs silently for 30–60+ min at this
step. The plan claimed ProgressLogger was wired into Stage B — this
internal loop was skipped.

**Root cause:** `way_pipeline.dart` (lines 181–216, pre-fix) iterated
every row of `ways_raw` via eager `db.select()` materialization, then
per-way per-node ran a prepared `SELECT 1 FROM nodes_raw WHERE id=?`
point lookup. O(N ways × M nodes/way) — ~40M SQLite calls on the full-
Germany extract, no ticker, no cadence emission.

**Fix (commit `47f554f`):**

1. **Streaming pass** — swapped `db.select()` for `stmt.selectCursor()`
   so ways are consumed one row at a time (bounded memory).
2. **Single JOIN instead of N×M point lookups** — inserted every
   `(way_id, node_id)` pair into a TEMP table `way_node_refs` inside a
   committed-every-100k-ways transaction, then ran ONE query:

   ```sql
   SELECT DISTINCT wn.way_id
   FROM way_node_refs wn
   LEFT JOIN nodes_raw n ON n.id = wn.node_id
   WHERE n.id IS NULL;
   ```

   Set-based query replaces ~40M point lookups. `DISTINCT` collapses
   ways with multiple missing refs to one dropped-row output (each way
   is dropped at most once, `filter_stats.deleted_node_ref` bumped
   once, `skipped.log` gets one entry).
3. **ProgressLogger tick on the streaming pass** — `total = COUNT(*)
   FROM ways_raw`, emits 5% / 5s progress lines.
4. **Extracted to testable top-level `runWayIntegrityCheck({scratch,
   onDrop})`** — semantics covered by 5 new unit tests
   (`test/filter/way_pipeline_test.dart`).

**Tests:** 239 → 244 sub-package tests green.
**Analyze:** clean.

### Gap 2 — Broken stdout pipe when shell died

**Symptom:** The Wave 5 executor ran
`dart run bin/osm_pipeline.dart ... > /c/tmp/germany-run.log 2>&1` from
bash. The harness eventually killed that shell (parent of the dart
subprocess), leaving `dartvm.exe` orphaned with a stdout pipe pointing
at a defunct parent. Subsequent log writes went into the void — the
pipeline appeared silent even though it was running.

**Fix (commit `9848a14`):** Added a `--log-file=<path>` CLI flag to
`bin/osm_pipeline.dart` that opens the log file INSIDE the Dart
process (so it survives regardless of what the invoking shell does).

* `lib/cli/logger.dart` gains a static `IOSink? _fileSink` + public
  `setFileSink()` / `fileSink` getter.
* `Logger.info/warn/error` now write to stderr AND (if attached) the
  file sink. Existing stdout output continues for interactive runs.
* `bin/osm_pipeline.dart` parses `--log-file`, opens the file BEFORE
  the first `Logger.*` call (so the banner + argument-parse errors
  land in the file too), and flushes/closes the sink in a `finally`
  block on BOTH the happy path and the `PipelineError` catch.

`ProgressLogger` inherits durability for free — it routes through
`Logger.info` via its default emit sink.

**Tests:** 4 new logger tests (`test/cli/logger_test.dart`):

1. Duplicates info/warn/error to the file sink.
2. Regression guard: stdout-only when file sink is null.
3. `setFileSink(null)` detaches without closing the caller-owned sink.
4. `fileSink` getter reflects attachment state.

**Manual smoke:**

* Happy path (`--pbf=test/fixtures/tiny.osm.pbf --log-file=/tmp/x.log`)
  produced a 65-line durable log capture — banner at top, `Pipeline
  OK.` at the bottom.
* Negative path (`--pbf=/nonexistent.pbf --log-file=/tmp/y.log`) wrote
  `[error] PBF file not found: …` to the log file before exiting with
  code 2 — the `finally` block correctly flushes on error paths.

**Sub-package tests after both fixes:** 248 total, all green. `dart
analyze` clean.
