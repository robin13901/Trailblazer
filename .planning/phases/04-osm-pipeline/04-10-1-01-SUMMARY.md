## Deviations from Plan

**None material.** Two minor implementation choices worth noting:

1. **Emit sink injection instead of stderr capture.** The plan said "redirect stderr in the test via `IOOverrides.runZoned` OR wrap Logger in a captured sink — pick whichever your project convention prefers." No prior test in `tool/osm_pipeline/test/` uses either pattern (grep-verified). Chose the sink-injection route: `ProgressLogger(..., emit: sink.add)`. Production defaults to `Logger.info`, so behavior is identical; tests avoid stderr-redirection ceremony. Class is now trivially testable without touching global state.

2. **Total-count SELECT is inline (not pre-computed).** For Stages D + E the plan noted `COUNT(*)` should be cheap. Confirmed on Berlin: no measurable delay. If Germany reveals slowness (unlikely — the `ways_raw` index handles it), we cache the count as a follow-up per the plan's Deviations §. Not needed at Berlin scale.

## Decisions Made (candidates for STATE.md)

- **D-04-10-1-01-A** — ProgressLogger's SendPort ingestion (`absorb(WorkerTick)`) lives in the emitter API from Wave 1. Wave 4 (Stage D isolates) wires workers → coordinator → `absorb(msg)` without refactoring the class. `WorkerTick(workerId, deltaTicks, elapsedMs)` is the marshalable message shape; `deltaTicks` is per-tick delta (not cumulative) so N workers contribute correctly to a single aggregate `_done` counter.
- **D-04-10-1-01-B** — `total=0` is the sentinel for "unknown total up front" — Stage B pass A, Stage C pass A, Stage F.1 water/labels PBF passes all use it. Emitter falls back to rate-only format (`{stage} progress: {done} {unit} — {rate}/s (total unknown)`).
- **D-04-10-1-01-C** — Tippecanoe stdout/stderr lines are pass-through with a `[Stage F.2]` prefix — no attempt to parse tippecanoe's own progress format. Verified end-to-end on Berlin: 60+ `[warn] [Stage F.2] …` lines from tippecanoe's own progress bar landed in the log correctly.
- **D-04-10-1-01-D** — Emit sink is injectable in the `ProgressLogger` constructor (`emit: void Function(String)?`, defaults to `Logger.info`). Enables clean unit tests without stderr redirection; keeps production output routed through the existing `Logger` API surface.
- **D-04-10-1-01-E** — `Logger` static class (23 lines) stays untouched. `ProgressLogger` is a sibling stateful helper in `lib/cli/progress_logger.dart`, not an extension of `logger.dart`. Rationale: many call sites depend on `Logger.info/warn/error`; migrating them all is not this plan's scope.
